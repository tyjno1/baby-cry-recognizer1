## 1. 🧩 项目背景

新生儿无法通过语言表达自身需求，主要依赖哭声传递信息。然而，传统“哭声解码”高度依赖父母或照顾者的经验，容易因误判导致反复尝试、情绪焦虑或照顾延误。现有婴儿监护设备多聚焦于生理体征监测（如呼吸、体温），缺乏对哭声语义的智能解析能力。随着AI音频处理与边缘计算技术的发展，利用机器学习对婴儿哭声进行分类，并实现自动化需求判断已成为可能。但市场上尚无成熟、可本地运行的桌面端解决方案，且多数系统缺乏适应性改进机制。因此，本项目旨在填补这一空白，为新手父母及专业照护人员提供一款能够实时解析哭声、持续优化判断准确率的智能辅助工具。

## 2. 🎯 核心目标

开发一套**可离线运行、具备自学习能力**的电脑端婴儿需求识别程序，实现以下两大核心突破：

- **实时解析**：通过麦克风捕获婴儿哭声，经AI音频特征提取，自动匹配至“吃、大便、小便、睡、玩”五种基本需求之一。
- **自适应进化**：引入用户反馈闭环机制，每次判断后允许用户纠正错误，系统自动记录并调整模型参数或规则权重，从而逐步提升个体化识别精度。

## 3. 💎 价值主张

**“听懂哭声，告别猜测——让每一次回应都准确、及时、安心”**

- **降低试错成本**：将传统“逐一排除”的照顾模式，转变为一步到位的精准响应，减少不必要的喂奶、换尿布等无效动作。
- **赋能新手父母**：尤其对缺乏经验的初为人父母者，提供科学的辅助决策，缓解焦虑，提升照护自信。
- **提升专业人员效率**：月嫂、育儿师可在多婴照护场景下快速定位需求，减少耗时观察。
- **可持续迭代**：用户反馈驱动的自适应机制使系统越用越准，数据积累可为后续研究（如早发性疾病预警）提供基础。

## 🏗️ 系统架构

### 模块划分

1. **桌面客户端（Electron + React）**  
   - 职责：提供跨平台的图形用户界面，负责用户交互（启动/停止录音、显示推理结果、接收反馈）、调用后端API、管理本地进程生命周期。  
   - 内部包含：UI组件（React + Ant Design）、音频采集管理（调用后端或本地getUserMedia）、状态管理（Redux/MobX）。

2. **后端核心服务（Python FastAPI）**  
   - 职责：作为本地HTTP服务，封装所有业务逻辑，包括音频处理、推理、反馈学习、持久化。  
   - 子模块：  
     - **音频采集模块**（PyAudio）：从麦克风实时捕获音频流，切分静音段，提取特征。  
     - **特征提取模块**（Librosa）：计算MFCC、过零率、基频等特征向量。  
     - **推理引擎模块**：  
       - 规则层：基于特征阈值的快速预判（如高音调+急促 → 饥饿）。  
       - 模型层：使用scikit-learn分类器（如随机森林、SVM），或ONNX Runtime加载预训练ONNX模型。  
     - **反馈学习模块**：根据用户纠正记录，执行增量训练或规则权重调整（如贝叶斯更新、模型部分拟合）。  
     - **数据管理模块**（SQLite + SQLAlchemy）：存储用户配置、反馈日志、模型参数、经验数据。

3. **本地数据库（SQLite）**  
   - 职责：持久化存储用户反馈记录、模型状态、错误日志、预设动作选项等。

4. **模型与经验数据目录**  
   - 职责：存放初始发布的ONNX模型文件、公开哭声经验数据（CSV/JSON格式），以及后续用户更新下载的模型文件。

---

### 数据流

```
用户点击“开始录音” 
   → Electron通过HTTP请求通知后端启动录音
   → 后端PyAudio捕获音频流，实时提取特征，缓存至内存
   → 用户点击“停止录音”或自动检测到哭声结束
   → 后端将特征向量送入推理引擎，输出需求类别（如“吃”）
   → 后端返回结果至Electron，前端展示给用户
   → 用户在UI上选择“正确”或“错误”（并给出实际需求）
   → 前端发送反馈记录（{音频特征ID, 系统判断, 实际需求, 时间戳}）至后端
   → 后端保存至SQLite，并触发增量学习/参数调整
   → 系统下次推理时使用更新后的模型
```

## 🔌 核心接口

以下接口均通过 FastAPI 暴露，Electron 通过 `http://localhost:8000` 访问。

### 1. 启动录音
- **路径**: `/api/audio/start`
- **方法**: `POST`
- **描述**: 通知后端开始从麦克风采集音频。后端返回一个 `session_id` 用于后续操作。
- **请求体**: 无（或可包含录音持续时间等参数）
- **响应示例**:
```json
{ "session_id": "abc123", "status": "recording" }
```

### 2. 停止录音并推理
- **路径**: `/api/audio/stop`
- **方法**: `POST`
- **描述**: 停止当前录音，对捕获的音频进行特征提取，调用推理引擎，返回需求类别及置信度。
- **请求体**:
```json
{ "session_id": "abc123" }
```
- **响应示例**:
```json
{
  "session_id": "abc123",
  "prediction": "hungry",
  "confidence": 0.85,
  "feature_id": "feat_001"
}
```

### 3. 提交用户反馈
- **路径**: `/api/feedback`
- **方法**: `POST`
- **描述**: 用户对某次推理结果进行纠正，提交实际需求。
- **请求体**:
```json
{
  "session_id": "abc123",
  "feature_id": "feat_001",
  "predicted_need": "hungry",
  "actual_need": "wet",
  "user_comment": "实际是尿布湿了"
}
```
- **响应示例**:
```json
{ "status": "recorded", "adjustment_applied": true }
```

### 4. 获取历史反馈记录
- **路径**: `/api/feedback/history`
- **方法**: `GET`
- **描述**: 分页查询用户的反馈历史，用于UI展示。
- **参数**: `?page=1&size=20`
- **响应示例**:
```json
{
  "total": 150,
  "items": [
    {
      "id": 1,
      "timestamp": "2025-04-01T10:00:00",
      "predicted": "hungry",
      "actual": "wet"
    }
  ]
}
```

### 5. 手动上传音频（离线处理）
- **路径**: `/api/audio/upload`
- **方法**: `POST`
- **描述**: 支持用户上传已录制的音频文件（WAV/MP3），返回推理结果。常用于批量测试或导入经验数据。
- **请求体**: `multipart/form-data` (file)
- **响应**: 同 `/api/audio/stop` 格式。

### 6. 当前模型状态
- **路径**: `/api/model/status`
- **方法**: `GET`
- **描述**: 返回当前使用的模型版本、准确率统计、规则权重等状态信息。
- **响应示例**:
```json
{
  "model_version": "v1",
  "total_feedback_count": 200,
  "overall_accuracy": 0.72,
  "last_update": "2025-04-01T12:00:00"
}
```

### 7. 系统健康检查
- **路径**: `/health`
- **方法**: `GET`
- **描述**: 用于 Electron 启动后检查后端是否就绪。
- **响应**: `{"status": "ok"}`

## ⚙️ 关键设计说明

- **进程管理**：Electron `main` 进程通过 `child_process.spawn` 启动 Python 后端（可打包为独立可执行文件或直接使用Python解释器），并监听后端端口的健康检查。在应用退出时优雅关闭子进程。
- **音频处理**：为了避免阻塞，音频采集在Python异步任务中进行，特征提取后立即释放流。推理请求采用同步请求模式，但整体设计支持后续切换到WebSocket实现实时流式推理。
- **模型自适应**：反馈记录存储在 `feedback_entries` 表，每次收到新反馈时，后台异步执行一次增量训练（如在线学习 `partial_fit` 或规则权重更新），更新后的模型保存至 `user_models/` 目录，并在下次推理时自动加载。
- **安全与隐私**：音频数据始终停留在本地，不上传云端。数据库文件路径位于用户数据目录（`app.getPath('userData')`）。

以上架构充分考虑了跨平台交付、AI推理、用户反馈闭环和轻量运维，可支撑初期 MVP 快速开发，并具备扩展至云端或移动端的能力。

## 1️⃣ 实体关系图（文字描述）

### 核心实体及其关系

- **`session`（录音会话）**：表示一次录音过程，包含录音的开始/结束时间、状态（如 recording、finished）、关联的音频文件路径等。
- **`audio_feature`（音频特征）**：存储从录音片段中提取的特征向量（如 MFCC 等），每个特征对应一个唯一 ID，并关联到所属的 `session`。
- **`inference_result`（推理结果）**：记录每次推理的输出，包括预测的需求类别、置信度，关联到 `audio_feature` 和 `session`。
- **`feedback`（用户反馈）**：用户对推理结果的纠正信息，关联到 `inference_result`，记录用户给出的实际需求类别及评论。
- **`need_category`（需求类别）**：预定义或用户自定义的婴儿需求类别（如 hungry、wet、sleepy 等），被 `inference_result` 和 `feedback` 引用。
- **`model_state`（模型状态）**：维护当前模型的元信息，包括版本号、总反馈数、准确率、最后更新时间、模型文件路径等。
- **`error_log`（错误日志）**：记录系统运行过程中的异常信息，用于调试和故障排查。

### 关系描述

```
session 1 ──── * audio_feature
session 1 ──── * inference_result（由一次录音可能产生多个特征分段，每个分段一次推理）
audio_feature 1 ──── 1 inference_result（一对一：一个特征对应一次推理结果）
inference_result 1 ──── * feedback（一次推理可能被多次纠正？实际单次只能有一条反馈，但设计成一对多以便扩展，当前使用唯一约束限制）
inference_result * ──── 1 need_category（预测类别）
feedback * ──── 1 need_category（实际类别）
```

## 4️⃣ 数据迁移策略

### 4.1 初始部署

- 使用 SQLAlchemy ORM 的 `Base.metadata.create_all(engine)` 在应用首次启动时自动创建所有表（如果不存在）。
- 同时插入默认的 `need_category` 数据（如 'hungry', 'wet', 'sleepy', 'uncomfortable', 'unknown' 等）。
- 创建第一条 `model_state` 记录，指向初始的 ONNX 模型文件（如 `onnx_model_v1.onnx`）。

### 4.2 版本升级（Schema 变更）

由于 SQLite 对 ALTER TABLE 支持有限（仅支持 ADD COLUMN 和 RENAME TABLE），建议采用以下轻量迁移策略：

1. **引入版本号表** `schema_version`：
   ```sql
   CREATE TABLE schema_version (
       id INTEGER PRIMARY KEY AUTOINCREMENT,
       version INTEGER NOT NULL,
       applied_at DATETIME DEFAULT CURRENT_TIMESTAMP
   );
   ```

2. **迁移脚本结构**：每个迁移定义为一个函数，包含：
   - 检查当前数据库版本（查询 `schema_version` 中最高版本）。
   - 如果低于目标版本，则执行对应 DDL（如创建新表、添加列、重建表等）。
   - 更新 `schema_version` 记录。

3. **兼容性处理**：对于需要修改列类型或删除列的操作，由于 SQLite 不支持，改用“创建新表 → 复制数据 → 删除旧表 → 重命名新表”的模式。

4. **示例迁移**（v1 → v2）：
   - v1 初始版本包含上述所有表。
   - v2 若需要给 `inference_result` 增加 `processing_time` 列，则执行：
     ```sql
     ALTER TABLE inference_result ADD COLUMN processing_time REAL;
     ```
   - 若需要删除字段，则采用“重建表”方式。

5. **集成到 FastAPI 启动过程**：在应用启动时调用迁移函数，确保数据库处于最新版本。

### 4.3 数据备份与回滚

- 由于是本地应用，可采用简单的文件备份：升级前将当前 SQLite 文件（如 `data.db`）重命名为 `data_backup_v1.db`。
- 若升级失败，应用可尝试恢复备份文件，并记录错误日志。

### 4.4 模型数据迁移

- 模型文件（ONNX 等）独立于数据库，存放在 `data/user_models/` 目录。
- 当新增模型版本时，插入新的 `model_state` 记录，`is_current` 更新为新版本，旧版本标记为 0。推理时根据 `is_current=1` 加载模型。

## 3. 认证授权方案

### 方案说明

由于是本地应用，采用轻量级认证方案：

### 3.1 Token生成与验证

```python
# 后端生成API Token
import secrets
import hashlib
from datetime import datetime, timedelta

class AuthManager:
    def __init__(self):
        self.api_key = self._generate_api_key()
        self.tokens = {}
    
    def _generate_api_key(self):
        """生成固定API密钥（安装时生成）"""
        return secrets.token_hex(32)
    
    def create_session_token(self):
        """创建临时会话令牌"""
        token = secrets.token_hex(16)
        self.tokens[token] = {
            'created_at': datetime.now(),
            'expires_at': datetime.now() + timedelta(hours=1)
        }
        return token
```

    -   将生成的安装包（`.exe`, `.dmg`）上传到更新服务器（如GitHub Releases，或自建的S3/OSS）。

8.  **用户更新**：用户打开应用时，Electron主进程的`autoUpdater`模块会下载`latest.yml`，与本地版本对比，如果有新版本，则提示用户下载并安装。

-   **Electron**：
    -   **启动**：`npm run electron:dev`。这个命令会同时启动后端服务、前端开发服务器和Electron窗口。Electron窗口会加载`localhost:3000`。

## 4. 监控方案

桌面应用的监控与云端服务不同，重点在于**本地日志**、**崩溃报告**和**用户行为分析**。

### 4.1. 本地日志

-   **后端**：使用Python的`logging`模块。配置`RotatingFileHandler`，按大小或时间滚动日志文件。日志级别设置为`INFO`（生产环境）或`DEBUG`（调试环境）。记录关键信息：
    -   API请求与响应（耗时、状态码）
    -   模型推理结果与置信度
    -   数据库操作
    -   异常堆栈信息
-   **前端**：使用`electron-log`库。它可以将日志输出到文件和控制台。记录：
    -   UI操作（用户点击了哪个按钮）
    -   API调用失败
    -   渲染进程错误
-   **日志存储**：统一存储在用户的应用数据目录下的`logs`文件夹中。

### 4.2. 崩溃报告

-   **工具**：集成`@sentry/electron`。Sentry提供了免费的开发者计划，非常适合开源或小型项目。
-   **实现**：
    -   在Electron主进程和渲染进程初始化时，配置Sentry DSN。
    -   当应用发生未捕获的异常（`uncaughtException`、`unhandledRejection`）时，Sentry会自动捕获并上报。
    -   上报信息包括：操作系统版本、应用版本、错误堆栈、设备信息等。
-   **优势**：开发者可以实时在Sentry后台看到用户遇到的崩溃，并快速定位问题。

### 4.3. 用户行为分析（可选，需告知用户）

-   **目的**：了解用户如何使用应用，哪些功能最受欢迎，从而指导产品迭代。
-   **工具**：`PostHog`（开源、自托管）或`Amplitude`。
-   **原则**：
    -   **匿名化**：不采集任何个人身份信息（PII），仅使用匿名ID。
    -   **透明度**：在应用首次启动时，明确弹窗告知用户“我们采集匿名使用数据以改进产品”，并提供关闭选项。
-   **采集内容**：
    -   用户点击“开始分析”按钮的次数。
    -   用户选择的需求类型分布。
    -   用户提供反馈的比例。
    -   应用启动次数和时长。

### 4.4. 健康检查与自愈

-   **Electron主进程**：定期（例如每30秒）向`http://localhost:8000/health`发送HTTP GET请求。
-   **后端FastAPI**：实现一个`/health`端点，返回`{"status": "ok"}`。如果请求失败或超时，主进程认为后端进程已崩溃，自动杀死旧进程并重新启动。
-   **日志轮转**：日志文件不能无限增长。使用`RotatingFileHandler`或`TimedRotatingFileHandler`，设置最大文件大小（如10MB）和备份数量（如保留最近5个文件）。

### 4.5. 模型性能监控

-   **指标**：跟踪模型推理的准确率、置信度分布、以及用户反馈的修正频率。
-   **实现**：在数据库中记录每一次推理的`prediction`、`confidence`、`user_feedback`（如果用户进行了修正）。后端可以定期运行一个分析脚本，生成一个报告（例如：“上周模型准确率为75%，较上月提升5%”）。这个报告可以存储在本地，或（在用户同意后）匿名上报给开发者。
